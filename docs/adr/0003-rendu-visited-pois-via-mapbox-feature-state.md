# Rendu des Visited POIs via Mapbox `feature-state` et tileset v2

Le rendu visité / non-visité des POIs et de leurs zones Voronoi sur la **Mobile App** passe par l'API Mapbox `setFeatureState`, alimentée par un `mvtId` numérique pré-calculé côté ETL et bake dans le tileset `pois-tileset-v2` comme native MVT feature id (recipe : `features.id: ["get", "mvtId"]`). Les expressions de style référencent `["feature-state", "visited"]` au lieu d'inliner la liste des **Visited POI** dans une expression `["match", ...]` ou `["in", ["literal", ...]]`.

Le `mvtId` est calculé par une fonction pure et déterministe `getMvtIdFromPoiId(poiId): number` (cf. `libs/shared-utils/src/utils/poi.ts`) appelée identiquement côté ETL (au moment du transform GeoJSON) et côté **Mobile App** (au moment du `setFeatureState`). Schéma de plages par préfixe OSM :

```
OSM-N-{id} → mvtId = id                  (0          .. 10^11)
OSM-W-{id} → mvtId = id + 10^11          (10^11      .. 2×10^11)
OSM-R-{id} → mvtId = id + 2 × 10^11      (2×10^11    .. 3×10^11)
```

Tous les `mvtId` restent sous `Number.MAX_SAFE_INTEGER` (≈ 9×10¹⁵), donc safe en JS Hermes et compatibles avec la spec MVT (`uint64`). Zéro collision par construction tant que les sources restent OSM uniquement.

Côté `<SymbolLayer>` POI, `iconImage` étant une layout property (interdit à `feature-state`), on double la couche : `pois-data-layer-v2` est rendu deux fois, une fois avec `iconImage="<cat>-color-v1"` et une fois avec `iconImage="<cat>-bw-v1"`, et `iconOpacity` (paint, autorisé à `feature-state`) bascule l'une et masque l'autre selon l'état visited de la feature.

## Considered Options

- **`feature-state` + tileset v2 avec `mvtId` numérique pré-encodé** — retenu.
- **`feature-state` + tileset v2 avec `features.id: ["get", "poiId"]`** (hash MTS string→uint64 côté serveur) — abandonné après probe in-app : le SDK Mapbox client ne refait pas le hash MTS, donc `setFeatureState(stringPoiId, ...)` ne matche aucune feature MVT (l'algo de hash MTS n'est pas documenté/reproductible côté client).
- **`feature-state` + patch `@rnmapbox/maps` pour exposer la prop `promoteId`** sur `<VectorSource>` — promoteId permettrait d'utiliser directement le `poiId` string comme id côté SDK. Écarté pour ne pas introduire un patch supplémentaire à maintenir à chaque upgrade rnmapbox.
- **Workaround `querySourceFeatures` + `setFeatureState(String(feat.id))` par idle** — fonctionne mais ré-introduit un délai d'affichage (état posé après le 1er paint des tuiles), coût JSI par event, et anti-pattern selon la doc Mapbox (feature-state est stocké par source, pas par tuile, donc à poser en amont).
- **Repli sur `["match", ["get", "poiId"], visitedPoiIds, ...]`** — hash O(1) côté Mapbox, sans modification de tileset, mais la liste de 14k IDs reste embarquée dans 3 expressions Voronoi paint (pression mémoire structurelle).
- **Overlay GeoJSON séparé via `<ShapeSource>`** — déjà testé pendant VG-470, OOM sur Android low-cost à 14k features (placement + collision detection).

## Why

- **Push the constant-K out of the JS-to-native bridge**. Une expression `["match"]` ou `["in", ["literal", ...]]` embarque la liste des **Visited POI** dans le style compilé. À K=14k, le style compilé pèse plusieurs centaines de Ko et est dupliqué dans chaque expression (1 sur POI `iconImage`, 3 sur Voronoi `fillColor`/`lineColor`/`fillOpacity`). `feature-state` extrait totalement K du style : chaque feature porte un état booléen côté natif Mapbox, le style ne réfère qu'à `["feature-state", "visited"]`.
- **Pré-encoder `mvtId` plutôt que se fier au hash MTS**. Le hash interne MTS qui convertit `poiId` string en `uint64` natif n'est pas exposé côté client. Le SDK Mapbox ne refait pas le hash quand on passe une string à `setFeatureState`. Pré-encoder un id numérique déterministe côté ETL (et le calculer identiquement côté mobile via une fonction partagée dans `shared-utils`) supprime toute ambiguïté et garantit que le matching client ↔ tileset est exact. Zéro dépendance à un algorithme opaque.
- **Pré-encoder plutôt que patch `promoteId` rnmapbox**. La prop `promoteId` n'est pas exposée par `@rnmapbox/maps@10.1.45` sur `<VectorSource>`. Un patch est faisable (la prop existe nativement côté `mapbox-maps-ios` et `mapbox-maps-android`) mais introduit une dette de maintenance à chaque upgrade rnmapbox. La voie pré-encodage est purement TS, hébergée dans le monorepo, n'introduit aucun fork natif.
- **Poser `setFeatureState` en amont plutôt qu'en réaction aux events de chargement**. Mapbox stocke le feature-state dans `SourceFeatureState`, une map par source clée par `(sourceLayer, featureId)` indépendante des tuiles chargées. Quand une tuile arrive (ou recharge après éviction), le SDK ré-applique automatiquement les entrées pendant `SourceCache.prepare()`, **avant** le premier paint de cette tuile. Conséquence : on n'a pas besoin de `querySourceFeatures` par `onMapIdle` / `onSourceDataLoaded`. On pose 10k`setFeatureState` une fois au mount + à chaque refetch, le SDK fait le reste. Pas de délai d'affichage entre l'arrivée des tiles et l'application du state.
- **Chunker via `setTimeout(0)`**. 10-14k `setFeatureState` fire-and-forget en un tick microtask peuvent saturer le bridge JSI sur Android low-end. Découper en batchs de 50 et yielder le JS thread via `await new Promise(r => setTimeout(r, 0))` entre chaque chunk laisse passer les autres tâches du loop pour ne pas bloquer la 1re frame interactive. Le chunk de 50 (plutôt que 500) est intentionnel : plus de yields = jank moins concentré sur Android low-end, et le coût d'overhead reste négligeable.
- **Versioning v2 plutôt qu'update in-place**. Le bump en `pois-tileset-v2` permet de garder `v1` actif tant que des versions de **Mobile App** déjà déployées le consomment. Cohérent avec ADR-0001 (versioning explicite par suffixe). À noter : les `sourceLayerID` consommés côté **Mobile App** (`pois-data-layer-v2`, `voronoi-zones-layer-v2`) sont également bumpés en v2 dans le même commit que le tileset, donc un rollback éventuel exige de revert le code mobile en plus de la variable d'environnement `EXPO_PUBLIC_POIS_MAPBOX_TILESET_URL` — un simple changement d'env ne suffit pas.
- **Double layer POI plutôt que repli `match`**. `match` tient en O(1) côté Mapbox mais la liste de 14k IDs reste embarquée dans 3 expressions Voronoi paint et est inlinée dans le style compilé : pression mémoire structurelle. `feature-state` supprime la pression mémoire **et** la recompile-de-style à chaque refetch du hook `useUserVisitedPois`. Le coût du double layer POI (deux symbol placements sur la même VectorSource) est bornable : `iconAllowOverlap=true` sur les deux couches sort le placement du jeu de collision global.

## Consequences

- **Côté ETL data-manager** : `apps/data-manager/src/etl/transform/pois.ts` et `generate-voronoi.ts` bakent `properties.mvtId = getMvtIdFromPoiId(poiId)` dans chaque feature GeoJSON. La recipe `apps/data-manager/src/etl/load/pois-recipe.json` porte `features.id: ["get", "mvtId"]` sur les deux layers. Une feature dont le `poiId` ne matche pas le format `OSM-{N|W|R}-{int}` fait `throw` côté ETL (fail-fast).
- **Côté shared-utils** : `getMvtIdFromPoiId(poiId): number` est la source de vérité du schéma de plages. Exporté depuis `libs/shared-utils`, consommé par `data-manager` et `mobile-app` pour garantir un mapping identique côté serveur et client.
- **Côté Mapbox Studio** : `glutomax.pois-tileset-v1` reste publié et actif tant que toutes les versions de **Mobile App** déployées (en particulier celles bloquées par mises à jour OTA en retard) le consomment. Suppression à planifier après quelques semaines.
- **Côté Mobile App code** : le hook `useApplyVisitedFeatureStates(mapRef, visitedPoiIds, isVisitedReady)` maintient `appliedMvtIdsRef` (`useRef<Set<number>>`), un Set qui reflète l'état effectivement posé sur le bridge Mapbox, mis à jour **chunk par chunk** (et non en bloc à la fin de passe). À chaque run du `useEffect`, il calcule `current = Set<mvtId>(visitedPoiIds)`, diff vs `appliedMvtIdsRef.current`, puis appelle `setFeatureState(String(mvtId), { visited: true }, ...)` pour `toAdd` et `setFeatureState(..., { visited: false })` pour `toRemove`, chunké par 50 via `setTimeout(0)`. Lifecycle : rendez-vous entre deux conditions async (style chargé + `isVisitedReady`) via `useState`. Le cleanup du `useEffect` flippe un flag `cancelled` lu **avant** chaque chunk pour interrompre une passe obsolète si `visitedPoiIds` change pendant qu'une boucle tourne. **Invariant clé** : parce que `appliedMvtIdsRef` est mis à jour après chaque chunk effectivement posé, une cancellation à mi-passe laisse le ref cohérent avec ce qui est réellement visited sur la map — la passe suivante calcule le diff vs cet état réel et termine le travail (les ids partiellement appliqués ne deviennent jamais orphelins). `onDidFinishLoadingStyle` signale (a) ; `isVisitedReady` (= `isSuccess` du React Query `useUserVisitedPois`) signale (b). Les composants `<MapPOILayers>` et `<MapVoronoiLayers>` ne reçoivent plus aucun id visited en prop — leur rendu repose intégralement sur `feature-state`. Le `symbolSortKey` POI_LABELS, layout property donc hors-portée de `feature-state`, ne porte plus de bonus visited (le tri reste piloté par la sélection et le filterLevel).
- **Pas de support `feature-state` pour les layout properties** : si on veut un jour piloter `iconImage`, `iconSize`, `symbolSortKey` ou `textField` selon l'état visited, il faudra soit doubler à nouveau la couche (comme on l'a fait pour `iconImage`), soit baker une propriété par feature côté tileset (impraticable pour un état utilisateur), soit accepter le coût O(K) d'un `match`.
- **Extension future à AI/CUSTOM** : l'enum `poiSourceEnum` côté DB prévoit déjà `OSM`, `AI`, `CUSTOM`. Quand l'une de ces sources sera peuplée, étendre `getMvtIdFromPoiId` avec de nouvelles plages au-dessus de `3 × 10^11` (par exemple via un hash sur le sourceId pour AI/CUSTOM qui n'ont pas d'id numérique natif), et mettre à jour cet ADR + la regex `POI_ID_PATTERN`. Tant que ce travail n'est pas fait, les POIs non-OSM ne pourront pas être visited.
- **Migration future vers `setFeatureStateExpression`** : Mapbox Maps SDK iOS+Android v11.17 (déc 2025) introduit une API expérimentale `setFeatureStateExpression()` qui pose des feature-states en bulk via une seule expression Mapbox. Quand `@rnmapbox/maps` embarquera une version Maps SDK ≥ 11.17, on pourra remplacer la boucle chunkée par un seul appel — réduction attendue de 1-2 ordres de grandeur sur le temps d'application en masse. La sémantique côté rendu est identique.
